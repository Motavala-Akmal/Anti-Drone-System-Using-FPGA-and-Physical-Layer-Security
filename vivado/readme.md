# Vivado Implementation

This folder contains the Vivado-related files, reports, screenshots, and implementation outputs for the FPGA-Based Anti-Drone Security System.

The Verilog HDL design is verified and analyzed using Xilinx Vivado. The Vivado workflow includes RTL design checking, behavioral simulation, waveform verification, RTL schematic generation, synthesis, utilization analysis, timing analysis, and power estimation.

## RTL Files Used

The main Verilog design files are stored in the `verilog/` directory of the repository.

Typical RTL files include:

```text
anti_drone_pls_core.v
anti_drone_pls_board_demo_top.v
pls_security_controller.v
top_wrapper.v
```

These files implement the FPGA-based Physical Layer Security controller, risk-score generation logic, state-based decision system, and board-level wrapper.

## Testbench Files Used

The testbench files are used only for simulation and verification.

Typical testbench files include:

```text
tb_anti_drone_pls_board_demo_top.v
tb_pls_security_controller.v
tb_top_wrapper.v
```

The testbench provides sample input scenarios such as:

* Normal / verified Bob condition
* Weak secrecy condition
* Suspicious activity condition
* Alarm condition
* Isolation condition

## Reports Folder

The `reports/` folder contains the main output reports generated from Vivado.

### `synthesis_report.pdf`

This report contains the synthesis summary of the Verilog design. It shows whether the RTL code was successfully converted into FPGA-level logic.

### `utilization_report.pdf`

This report shows the FPGA resource usage, such as LUTs, flip-flops, I/O pins, and other hardware resources used by the design.

### `timing_report.pdf`

This report shows the timing behavior of the design. It helps verify whether the design can meet the required clock constraints.

### `power_report.pdf`

This report shows the estimated power consumption of the FPGA design. It is useful for understanding the hardware efficiency of the controller.

## Main Output Signals Observed

The following outputs are mainly observed during Vivado simulation:

| Output Signal     | Description                                  |
| ----------------- | -------------------------------------------- |
| `risk_score`      | Final calculated threat score                |
| `decision_code`   | Encoded decision generated by the controller |
| `pls_mode`        | Physical Layer Security operating mode       |
| `normal_led`      | Indicates normal or verified condition       |
| `secure_led`      | Indicates secure monitoring mode             |
| `alarm_led`       | Indicates alarm condition                    |
| `isolate_led`     | Indicates isolation condition                |
| `relay_trip`      | Indicates relay disconnection or trip signal |
| `charging_enable` | Enables or disables the protected operation  |

## PLS Mode Mapping

| PLS Mode | Meaning                |
| -------- | ---------------------- |
| `00`     | Normal mode            |
| `01`     | Secure monitoring mode |
| `10`     | Alarm mode             |
| `11`     | Isolation mode         |

---

## Decision Code Mapping

| Decision Code | Meaning                          |
| ------------- | -------------------------------- |
| `0`           | Normal / verified Bob / low risk |
| `1`           | Needs verification               |
| `2`           | Suspicious / secure monitoring   |
| `3`           | Likely Eve / alarm               |
| `4`           | High-risk Eve / isolation        |

## Notes

Large auto-generated Vivado folders should not be uploaded to GitHub because they can make the repository unnecessarily large and difficult to manage.

Avoid uploading folders such as:


Only important reports, screenshots, selected project notes, and manually written source files should be included.
